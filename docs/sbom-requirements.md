# SBOM Requirements

Exploit Intelligence supports two SBOM formats for analysis requests:

| Format | Use case |
|--------|----------|
| **SPDX 2.3** | Multi-component **product** analysis 
| **CycloneDX 1.6** | **Single-component** analysis 

## SPDX 2.3 (Multi-component)

### Example

See [`gitops-1.19.json`](../src/test/resources/devservices/spdx-sboms/gitops-1.19.json) for a complete SPDX 2.3 document that follows the structure below.

### Document structure

An SPDX product SBOM MUST:

- Set `spdxVersion` to `SPDX-2.3`.
- Include a `DESCRIBES` relationship from the document to a **product** package (the product name is taken from that package).
- List **component** packages linked to the product with `PACKAGE_OF` relationships.
- Give each component an OCI package URL (`purl`) in `externalRefs` with `referenceCategory` `PACKAGE_MANAGER` and `referenceType` `purl`. Supported components use the form:

  `pkg:oci/<name>@sha256:<digest>?repository_url=<registry>/<path>&tag=<tag>`

At least one component with a valid OCI `purl` is required; components without an OCI `purl` are excluded from analysis.

Optionally, the product package MAY include a CPE in `externalRefs` (`referenceCategory` `SECURITY`, `referenceType` `cpe22Type`). If the CPE exists, it will be stored in the product metadata.

### Container image labels

The SPDX file references container images by OCI `purl`; it does not embed source-code metadata itself. For each image referenced by a component, the image MUST carry OCI labels that identify the Git source repository and commit (or ref) used to build it. During processing, Syft scans each image and reads these labels; missing labels cause that component to fail validation and be excluded from analysis.

Each image MUST include:

| Purpose | Preferred OCI label | Accepted alternatives (deployment-configurable) |
|--------|---------------------|--------------------------------------------------|
| Source repository URL | `org.opencontainers.image.source` | `image.source-location`, `io.openshift.build.source-location`, `upstream-source-url` |
| Source commit / ref | `org.opencontainers.image.revision` | `image.source.commit-id`, `io.openshift.build.commit.id`, `upstream-source-ref` |

---

## CycloneDX 1.6 (single component)

### Example

See [`nmstate-rhel8-operator.json`](../src/test/resources/devservices/cyclonedx-sboms/nmstate-rhel8-operator.json) for a complete CycloneDX 1.6 document produced by Syft from a container image scan. It includes the required `metadata.component`, source-code properties, and a populated `components` list of packages found in the image.

### Document structure

A CycloneDX single-component SBOM MUST:

- Set `bomFormat` to `CycloneDX` and `specVersion` to `1.6`.
- Include `metadata.component` with a non-empty `name` (typically the container image reference). Optional fields such as `type`, `version`, `bom-ref`, and `purl` are copied to report metadata when present.
- Include a `components` array listing the packages discovered in the SBOM (for example, RPMs or libraries from a Syft image scan).

### Source-code metadata

Unlike SPDX, CycloneDX carries source-code metadata directly in the document. The backend reads `metadata.properties` and requires entries for the Git repository URL and commit (or ref) of the analyzed component.

Each property is a `{ "name": "...", "value": "..." }` pair in `metadata.properties`. The backend matches property names against the configured key lists (same labels as for container images):

| Purpose | Preferred property name | Accepted alternatives (deployment-configurable) |
|--------|-------------------------|--------------------------------------------------|
| Source repository URL | `org.opencontainers.image.source` | `image.source-location`, `io.openshift.build.source-location`, `upstream-source-url` |
| Source commit / ref | `org.opencontainers.image.revision` | `image.source.commit-id`, `io.openshift.build.commit.id`, `upstream-source-ref` |

When the SBOM is generated by Syft from a container image, OCI labels are copied into `metadata.properties` with a `syft:image:labels:` prefix (for example `syft:image:labels:io.openshift.build.source-location`). The backend accepts properties with or without that prefix, so both hand-authored and Syft-generated SBOMs work.

The nmstate example shows this pattern: `metadata.properties` includes `syft:image:labels:io.openshift.build.source-location` and `syft:image:labels:io.openshift.build.commit.id`, which satisfy the source URL and commit ID requirements.
